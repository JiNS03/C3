## 0. 자동화 기초 개념

### Trigger (트리거)
어떤 일이 일어나면 자동화를 시작한다는 조건. 자동화 워크플로우의 **출발점**.

### Action (액션)
Trigger가 발동한 뒤 자동으로 수행되는 처리 동작.

### 조건 분기 (Filter, Router)

| 구분 | 설명 |
|---|---|
| **Filter (Zapier)** | 조건을 만족하면 통과, 조건 불충족 시 중단하는 방식. 경로가 단 하나 |
| **Router (Make)** | 조건에 따라 여러 경로로 분기시키는 방식. 하나의 흐름이 2개 이상의 길로 갈라짐 |

# 프로젝트 1. 자동화 도구 비교 구현

## 1. 개요

- **워크플로우 주제**: 지원서 접수 자동 분류 및 알림
- **비교 도구**: Make, Zapier
- **워크플로우 요약**: Google Form으로 지원서를 접수받아, 응답 데이터가 쌓이는 Google Sheets를 감지(Trigger)하고, 희망 분야(기획팀/개발팀)에 따라 조건 분기(Router/Paths)한 뒤, 각 분야별 시트에 자동으로 행을 기록(Action)하는 워크플로우

## 2. 워크플로우 구조

```
Trigger: Google Sheets 새 행 감지 (설문지 응답 시트1)
     ↓
조건 분기: 희망 분야 (기획팀 / 개발팀)
     ├─ 기획팀 → Action: "기획팀" 시트에 행 추가
     └─ 개발팀 → Action: "개발팀" 시트에 행 추가
```

- **Trigger**: 1개 (Google Sheets 새 행 감지)
- **Action**: 2개 (기획팀 시트 기록, 개발팀 시트 기록)
- **조건 분기**: 1개 (희망 분야 기준 2갈래 분기)
- 두 분기 경로 모두 실제 지원서 제출(기획팀 1건, 개발팀 1건)을 통해 각각 1회 이상 실행 확인 완료

---

## 3. 도구별 구현

### 3-1. Make 구현

- **Trigger**: `Google Sheets – Watch New Rows` (설문지 응답 시트1)
  - 처음엔 `Google Forms – Watch Responses`로 시도했으나, 응답 데이터가 Collection 구조로 중첩되어 있어 실제 텍스트 값(이름, 희망 분야 등)을 꺼내올 수 없는 문제가 발생하여 Google Sheets 직접 감지 방식으로 변경
- **조건 분기**: `Router` 모듈 사용, 경로 2개(기획팀/개발팀)로 분기
  - 필터 조건: `희망 분야` Equal to `기획팀` / `개발팀`
- **Action**: `Google Sheets – Add a Row` (기획팀 시트 / 개발팀 시트 각각 연결)
- **실행 시작 시점**: "Since specific ID" 옵션으로 기존 응답은 제외하고, 설정 시점 이후의 새 행만 감지하도록 구성
- **자동 실행**: 시나리오 하단 스케줄 토글을 켜서(ON) Trigger 발생 시 자동 실행되도록 설정


### 3-2. Zapier 구현

- **Trigger**: `Google Sheets – New or Updated Spreadsheet Row` (설문지 응답 시트1, 기준 열: 타임스탬프)
  - Make와 달리 처음부터 Google Sheets 기반 Trigger를 사용하여, Google Forms 특유의 복잡한 데이터 구조 문제 없이 단순하게 필드 값을 바로 가져올 수 있었음
- **조건 분기**: `Paths by Zapier` 사용, Path A(기획팀) / Path B(개발팀) 2갈래로 분기
  - 조건: `희망 분야` (Text) Contains `기획팀` / `개발팀`
  - Zapier 무료 플랜에서도 Paths 기능 사용 가능함을 확인
- **Action**: `Google Sheets – Create Spreadsheet Row` (Path A → 기획팀 시트 / Path B → 개발팀 시트)
- **자동 실행**: Zap을 "Publish"하여 활성화, 이후 실제 폼 제출 시 자동 실행 및 Zap History에서 실행 기록 확인


### 3-3. 공통 결과 확인

- 기획팀 지원자 제출 1건 → 양쪽 도구 모두 "기획팀" 시트에 정상 기록 확인
- 개발팀 지원자 제출 1건 → 양쪽 도구 모두 "개발팀" 시트에 정상 기록 확인

## 4. 비교 분석

| 비교 항목 | Make | Zapier |
|---|---|---|
| **UI/UX** | 캔버스에 모듈을 시각적 노드(원형 아이콘)로 배치하고 선으로 연결하는 방식. 전체 흐름을 한눈에 그림처럼 파악하기 좋음 | 위에서 아래로 순서대로 쌓이는 리스트/카드 형태. 단계별 설정 화면이 구조화되어 있어 순차적으로 따라가기 편함 |
| **설정 난이도** | Google Forms 연동 시 응답 데이터가 Collection(중첩 구조)으로 제공되어 실제 값을 꺼내는 과정이 다소 복잡했음. Google Sheets Trigger로 전환 후에는 단순해짐 | Google Sheets Trigger 기준으로는 필드가 바로 단순하게 노출되어, 초반부터 매핑이 직관적이었음 |
| **연동 서비스 범위** | Google Forms, Google Sheets, Slack 등 광범위한 연동 지원. 단, 일부 앱은 데이터 구조가 복잡하게 제공될 수 있음 | Google Sheets, Slack, Gmail 등 광범위한 연동 지원. 실제 사용 결과 Sheets 연동은 필드 매핑이 더 단순했음 |
| **무료 플랜 범위** | 월 1,000 Ops(작업 수) 제공 | 월 100 Tasks(작업 수) 제공. Paths(조건 분기) 기능도 무료 플랜에서 사용 가능함을 확인 |
| **실행 로그 확인 방식** | 캔버스 상의 각 모듈 위 작은 배지(숫자/체크 아이콘)를 클릭하면 Input/Output 데이터를 바로 확인 가능 | 별도의 "Zap History" 메뉴에서 실행 기록을 리스트 형태로 확인, 각 단계별 성공/실패 여부 확인 가능 |
| **조건 분기 기능** | Router 모듈로 여러 경로 분기, 무료 플랜에서 기본 제공 | Paths 기능으로 여러 경로 분기, 무료 플랜에서도 사용 가능 (사전 우려와 달리 제한 없었음) |

## 5. 각 도구의 장단점

### Make

**장점**
- 워크플로우 전체 구조를 시각적으로 한눈에 파악하기 좋음
- 무료 플랜의 작업 처리량(1,000 Ops)이 Zapier보다 넉넉함
- 각 모듈의 실행 데이터를 캔버스 위에서 바로 확인 가능해 디버깅이 편리함

**단점**
- Google Forms처럼 데이터 구조가 복잡한 서비스 연동 시, 실제 값을 꺼내는 과정이 직관적이지 않을 수 있음
- 초기 학습 곡선이 다소 있음 (모듈 간 관계, Collection 구조 이해 필요)

### Zapier

**장점**
- 단계별 설정 흐름(Setup → Configure → Test)이 명확하게 구조화되어 있어 처음 사용해도 따라가기 쉬움
- Google Sheets 등 기본 서비스 연동 시 필드 매핑이 단순하고 직관적임
- Paths(조건 분기) 기능을 무료 플랜에서도 사용할 수 있어 접근성이 좋음

**단점**
- 무료 플랜의 작업 처리량(100 Tasks)이 Make보다 제한적임
- 실행 로그를 확인하려면 별도 메뉴(Zap History)로 이동해야 해서, Make처럼 캔버스에서 바로 확인하는 것보다 한 단계 더 필요함

---

## 6. 어떤 상황에 적합한가 (의견)

- **Make가 적합한 경우**: 여러 단계와 복잡한 분기가 얽힌 워크플로우를 한눈에 조망하며 설계하고 싶을 때, 그리고 무료 플랜에서 더 많은 작업량이 필요한 경우에 적합하다고 생각한다.
- **Zapier가 적합한 경우**: 자동화를 처음 접하는 사용자가 단계별로 명확한 가이드를 따라가며 빠르게 워크플로우를 완성하고 싶을 때, 그리고 Google Sheets 등 단순한 데이터 소스를 다룰 때 특히 편리하다고 생각한다.

이번 과제를 통해 동일한 목적의 워크플로우라도 도구에 따라 데이터를 다루는 방식과 사용자 경험이 다르다는 점을 직접 확인할 수 있었다.

# 프로젝트 2. 자유 주제 자동화 설계 및 구현

## 8. 자동화할 반복 업무 정의

**반복 업무**: 매일 여러 통씩 들어오는 이메일(문의/알림 메일 등)을 하나씩 열어 내용을 확인하고, 핵심 내용을 정리해두는 작업

**문제 상황**: 메일 한 통 확인하고 요약하는 데 몇 분이 걸린다고 해도, 하루에 여러 통이 쌓이면 반복 작업만으로 상당한 시간이 소모됨. 특히 "긴급"으로 분류되어야 할 메일과 일반 메일을 구분해서 대응 속도를 다르게 가져가야 하는 점도 매번 사람이 판단해야 하는 번거로움이 있음.

**자동화 목표**: 새 이메일이 도착하면, 제목에 "긴급" 포함 여부로 자동 분류하고, AI가 본문을 요약한 뒤, 분류에 맞는 제목을 붙여 요약 메일이 자동 발송되도록 한다.

## 9. 도구 선정 및 선정 이유

**선정 도구**: Make

**선정 이유**:
- 프로젝트 1에서 이미 Make로 Router(조건 분기)와 Google 서비스 연동을 다뤄본 경험이 있어, 동일한 환경에서 AI 모듈까지 확장하기 수월함
- Make는 Google Gemini AI 모듈이 기본 내장되어 있어, 별도의 복잡한 연동 설정 없이 "AI 요약 Action"을 추가할 수 있음
- Gemini는 무료 등급(Free tier) API Key 발급이 가능해, 소규모 테스트 용도로는 별도 비용 없이 사용 가능함 (과금 리스크 완화)
- Gmail은 이미 Google 계정 연결이 되어 있어 별도의 워크스페이스/채널 설정 없이 바로 발송 Action을 구성할 수 있어 가장 간단함
- 모듈 단위로 **"Add error handler"** 기능을 붙일 수 있어, 실패 알림 및 대체 경로(보너스2) 설계가 구조적으로 명확함

## 10. 워크플로우 설계

### 전체 구조

```
Trigger: Gmail – Watch emails (라벨 "요약대상" 붙은 메일 감지)
     ↓
조건 분기(Router): 제목에 "긴급" 포함 여부
     ├─ 긴급   ─ Action 1: Google Gemini AI – 본문 3줄 요약 생성
     │           Action 2: Gmail – "[긴급] 메일 요약" 발송
     └─ 긴급 아님 ─ Action 1: Google Gemini AI – 본문 3줄 요약 생성
                    Action 2: Gmail – "[일반] 메일 요약" 발송

[보너스2, 계획] 각 Gmail 발송 Action 실패 시
     → 에러 핸들러: 담당자에게 별도 실패 알림 메일 발송 + 백업 시트에 임시 기록
```

### 단계별 흐름 설명

1. **Trigger** — Gmail에서 **"요약대상"** 라벨이 붙은 메일을 `Watch emails` 모듈(Gmail 필터, Query: `label:요약대상`)로 감지
2. **조건 분기(Router)** — 감지된 메일의 제목(Subject)에 **"긴급"**이라는 단어가 포함되어 있는지 여부로 두 갈래로 분기
   - **1st 경로(긴급)**: `Subject` Contains `긴급`
   - **2nd 경로(일반)**: `Subject` Does not contain `긴급`
3. **Action 1 – AI 요약 [보너스1]** — 각 경로에서 **Google Gemini AI**(Generate a response) 모듈을 통해 메일 본문(Full text body)을 3줄로 요약하는 프롬프트를 실행
4. **Action 2 – Gmail 발송** — 발신자, 원본 제목, AI 요약 결과를 조합한 내용을 담아 지정한 수신자에게 이메일 발송
   - 긴급 경로: 제목 `[긴급] 메일 요약`
   - 일반 경로: 제목 `[일반] 메일 요약`
5. **에러 핸들러 [보너스2, 계획]** — Gmail 발송 단계에서 오류가 발생할 경우, 시나리오가 멈추지 않고 계속 진행되도록 "Resume" 처리하면서, 담당자에게 별도의 실패 알림 메일을 보내고, 실패한 메일 정보를 백업용 스프레드시트에 임시로 기록하여 데이터 유실을 방지 (다음 단계로 구현 예정)

- **Trigger**: 1개 (Gmail Watch emails)
- **Action**: 2개 (Google Gemini AI 요약, Gmail 발송)
- **조건 분기**: 1개 (제목 내 "긴급" 포함 여부, Router 사용)
- 긴급/일반 두 경로 모두 실제 테스트 메일로 각 1회 이상 실행하여 정상 동작 확인 완료

### 자동 실행 구조

Make 시나리오 하단의 스케줄 토글을 **ON**으로 설정하여, Gmail Trigger가 주기적으로 새 메일을 확인하고 조건에 맞으면 자동으로 전체 흐름이 실행되도록 구성 예정. (Trigger 발생 시 사람의 개입 없이 자동 실행되도록 마무리 작업 진행 중)


## 11. 구현 및 실행 결과

### 구현 과정 요약

1. Gmail **Watch emails** Trigger 추가 → Filter type을 "Gmail 필터"로 변경 후 Query에 `label:요약대상` 입력하여, 요약 대상으로 라벨링된 메일만 감지하도록 구성
2. **Router** 추가 → 제목의 "긴급" 포함 여부로 1st(긴급)/2nd(일반) 두 경로로 분기
3. 각 경로에 **Google Gemini AI(Generate a response)** 모듈 추가 → API Key 연결 후, 메일 본문을 3줄 요약하는 프롬프트 구성
4. 각 경로에 **Gmail(Send an email)** 모듈 추가 → Body type을 "Raw HTML"로 설정하고, 발신자/원본 제목/AI 요약 결과를 조합한 내용을 작성
5. 테스트 메일 2건(긴급 1건, 일반 1건)을 직접 작성해 발송하고 "요약대상" 라벨을 붙인 뒤, Trigger의 "Choose where to start"를 "All"로 설정하여 기존 메일부터 처리되도록 조정
6. **Run once**로 전체 흐름 실행 → 두 경로 모두 정상적으로 AI 요약 생성 및 메일 발송까지 완료됨을 확인



## 12. 보너스 과제 반영 내역

### 보너스 1 – AI 연동 Action 추가 (완료)
워크플로우의 Action 1 단계에서 **Google Gemini AI** 모듈(Generate a response)을 사용해, 감지된 메일의 본문을 3줄로 요약하는 텍스트 생성 Action을 구현함. 긴급/일반 두 경로 모두 각각 Gemini 모듈을 연결하여, 분기된 이후에도 AI 요약이 정상 동작함을 확인함.

> 참고: Google Gemini API는 무료 등급(Free tier)으로 발급 및 사용이 가능해, 별도의 유료 결제 없이 보너스 과제를 완수할 수 있었음. 다만 모델 사용량에 따라 소요 시간의 변화가 있으며, 토큰 소진 시 사용 불가능.

### 보너스 2 – 실패 알림 및 재시도 전략
Gmail 발송 Action에 에러 핸들러를 연결하여, 발송 실패 시:
1. 담당자에게 별도의 이메일로 실패 알림 발송
2. 실패한 메일 정보를 메일 본문에 삽입
3. 에러 핸들러 유형을 "Resume"으로 설정하여 시나리오 전체가 중단되지 않도록 처리
