Trigger : 어떤 일이 일어나면 자동화를 시작한다는 조건. 자동화 워크플로우의 출발점
Action : Trigger가 발동한 뒤 자동으로 수행되는 처리 동작

조건 분기 (Filter, Router)
- Filter(Zapier) : 조건을 만족하면 통과, 조건 불충족 시 중단하는 방식, 길이 단 하나
- Router(Make) : 조건에 따라 서로 여러 경로로 분기시키는 방식, 하나의 흐름이 2개 이상의 길로 갈라짐
  


# 자동화 도구 비교 구현 보고서
### 코디세이 3번째 미션 - 프로젝트 1

---

## 1. 개요

- **워크플로우 주제**: 지원서 접수 자동 분류 및 알림
- **비교 도구**: Make, Zapier
- **워크플로우 요약**: Google Form으로 지원서를 접수받아, 응답 데이터가 쌓이는 Google Sheets를 감지(Trigger)하고, 희망 분야(기획팀/개발팀)에 따라 조건 분기(Router/Paths)한 뒤, 각 분야별 시트에 자동으로 행을 기록(Action)하는 워크플로우

## 2. 워크플로우 구조

```
Trigger: Google Sheets 새 행 감지 (설문지 응답 시트1)
     ↓
조건 분기: 희망 분야 (기획팀 / 개발팀)
     ├─ 기획팀 → Action: "기획팀" 시트에 행 추가
     └─ 개발팀 → Action: "개발팀" 시트에 행 추가
```

- **Trigger**: 1개 (Google Sheets 새 행 감지)
- **Action**: 2개 (기획팀 시트 기록, 개발팀 시트 기록)
- **조건 분기**: 1개 (희망 분야 기준 2갈래 분기)
- 두 분기 경로 모두 실제 지원서 제출(기획팀 1건, 개발팀 1건)을 통해 각각 1회 이상 실행 확인 완료

---

## 3. 도구별 구현

### 3-1. Make 구현

- **Trigger**: `Google Sheets – Watch New Rows` (설문지 응답 시트1)
  - 처음엔 `Google Forms – Watch Responses`로 시도했으나, 응답 데이터가 Collection 구조로 중첩되어 있어 실제 텍스트 값(이름, 희망 분야 등)을 꺼내올 수 없는 문제가 발생하여 Google Sheets 직접 감지 방식으로 변경
- **조건 분기**: `Router` 모듈 사용, 경로 2개(기획팀/개발팀)로 분기
  - 필터 조건: `희망 분야` Equal to `기획팀` / `개발팀`
- **Action**: `Google Sheets – Add a Row` (기획팀 시트 / 개발팀 시트 각각 연결)
- **실행 시작 시점**: "Since specific ID" 옵션으로 기존 응답은 제외하고, 설정 시점 이후의 새 행만 감지하도록 구성
- **자동 실행**: 시나리오 하단 스케줄 토글을 켜서(ON) Trigger 발생 시 자동 실행되도록 설정

### 3-2. Zapier 구현

- **Trigger**: `Google Sheets – New or Updated Spreadsheet Row` (설문지 응답 시트1, 기준 열: 타임스탬프)
  - Make와 달리 처음부터 Google Sheets 기반 Trigger를 사용하여, Google Forms 특유의 복잡한 데이터 구조 문제 없이 단순하게 필드 값을 바로 가져올 수 있었음
- **조건 분기**: `Paths by Zapier` 사용, Path A(기획팀) / Path B(개발팀) 2갈래로 분기
  - 조건: `희망 분야` (Text) Contains `기획팀` / `개발팀`
  - Zapier 무료 플랜에서도 Paths 기능 사용 가능함을 확인
- **Action**: `Google Sheets – Create Spreadsheet Row` (Path A → 기획팀 시트 / Path B → 개발팀 시트)
- **자동 실행**: Zap을 "Publish"하여 활성화, 이후 실제 폼 제출 시 자동 실행 및 Zap History에서 실행 기록 확인

## 4. 비교 분석

| 비교 항목 | Make | Zapier |
|---|---|---|
| **UI/UX** | 캔버스에 모듈을 시각적 노드(원형 아이콘)로 배치하고 선으로 연결하는 방식. 전체 흐름을 한눈에 그림처럼 파악하기 좋음 | 위에서 아래로 순서대로 쌓이는 리스트/카드 형태. 단계별 설정 화면이 구조화되어 있어 순차적으로 따라가기 편함 |
| **설정 난이도** | Google Forms 연동 시 응답 데이터가 Collection(중첩 구조)으로 제공되어 실제 값을 꺼내는 과정이 다소 복잡했음. Google Sheets Trigger로 전환 후에는 단순해짐 | Google Sheets Trigger 기준으로는 필드가 바로 단순하게 노출되어, 초반부터 매핑이 직관적이었음 |
| **연동 서비스 범위** | Google Forms, Google Sheets, Slack 등 광범위한 연동 지원. 단, 일부 앱은 데이터 구조가 복잡하게 제공될 수 있음 | Google Sheets, Slack, Gmail 등 광범위한 연동 지원. 실제 사용 결과 Sheets 연동은 필드 매핑이 더 단순했음 |
| **무료 플랜 범위** | 월 1,000 Ops(작업 수) 제공 | 월 100 Tasks(작업 수) 제공. Paths(조건 분기) 기능도 무료 플랜에서 사용 가능함을 확인 |
| **실행 로그 확인 방식** | 캔버스 상의 각 모듈 위 작은 배지(숫자/체크 아이콘)를 클릭하면 Input/Output 데이터를 바로 확인 가능 | 별도의 "Zap History" 메뉴에서 실행 기록을 리스트 형태로 확인, 각 단계별 성공/실패 여부 확인 가능 |
| **조건 분기 기능** | Router 모듈로 여러 경로 분기, 무료 플랜에서 기본 제공 | Paths 기능으로 여러 경로 분기, 무료 플랜에서도 사용 가능 (사전 우려와 달리 제한 없었음) |

## 5. 각 도구의 장단점

### Make

**장점**
- 워크플로우 전체 구조를 시각적으로 한눈에 파악하기 좋음
- 무료 플랜의 작업 처리량(1,000 Ops)이 Zapier보다 넉넉함
- 각 모듈의 실행 데이터를 캔버스 위에서 바로 확인 가능해 디버깅이 편리함

**단점**
- Google Forms처럼 데이터 구조가 복잡한 서비스 연동 시, 실제 값을 꺼내는 과정이 직관적이지 않을 수 있음
- 초기 학습 곡선이 다소 있음 (모듈 간 관계, Collection 구조 이해 필요)

### Zapier

**장점**
- 단계별 설정 흐름(Setup → Configure → Test)이 명확하게 구조화되어 있어 처음 사용해도 따라가기 쉬움
- Google Sheets 등 기본 서비스 연동 시 필드 매핑이 단순하고 직관적임
- Paths(조건 분기) 기능을 무료 플랜에서도 사용할 수 있어 접근성이 좋음

**단점**
- 무료 플랜의 작업 처리량(100 Tasks)이 Make보다 제한적임
- 실행 로그를 확인하려면 별도 메뉴(Zap History)로 이동해야 해서, Make처럼 캔버스에서 바로 확인하는 것보다 한 단계 더 필요함

## 6. 어떤 상황에 적합한가 (의견)

- **Make가 적합한 경우**: 여러 단계와 복잡한 분기가 얽힌 워크플로우를 한눈에 조망하며 설계하고 싶을 때, 그리고 무료 플랜에서 더 많은 작업량이 필요한 경우에 적합하다고 생각한다.
- **Zapier가 적합한 경우**: 자동화를 처음 접하는 사용자가 단계별로 명확한 가이드를 따라가며 빠르게 워크플로우를 완성하고 싶을 때, 그리고 Google Sheets 등 단순한 데이터 소스를 다룰 때 특히 편리하다고 생각한다.

이번 과제를 통해 동일한 목적의 워크플로우라도 도구에 따라 데이터를 다루는 방식과 사용자 경험이 다르다는 점을 직접 확인할 수 있었다.
